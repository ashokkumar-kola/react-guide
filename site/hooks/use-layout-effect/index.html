    useLayoutEffect

    useLayoutEffect fires synchronously after DOM updates but before the browser paints the screen.
    It allows you to read layout, measure DOM elements, and block the browser from painting until your logic runs.

    It behaves like useEffect but with timing differences.

    1. üìù Syntax

    useLayoutEffect(() => {
  // run after DOM updates, before paint
  return () => {
    // cleanup
  };
}, [dependencies]);

    2. üéØ When to Use

    Use useLayoutEffect when you need:

    1. DOM measurements

    2. width, height, bounding boxes

    3. Synchronous layout reads + updates

    4. measuring layout before the browser paints

    5. Prevent visual flicker

    6. updating styles before paint

    7. Mutating DOM elements

    8. scroll position
    9. focus
    10. manually applying CSS updates

    3. üîß Timing Difference: useEffect vs useLayoutEffect

    useEffect

    • Runs after paint
    • Does not block the browser
    • Best for async logic, subscriptions, API calls

    useLayoutEffect

    • Runs before paint
    • Blocks painting
    • Perfect for layout-sensitive logic

    Timeline

    Render ‚Üí DOM Updated ‚Üí useLayoutEffect ‚Üí Browser Paint ‚Üí useEffect

    4. üìè DOM Measurement Example

    function Box() {
  const boxRef = useRef(null);
  const [width, setWidth] = useState(0);

  useLayoutEffect(() => {
    const rect = boxRef.current.getBoundingClientRect();
    setWidth(rect.width);
  }, []);

  return (
    <>
      <div ref={boxRef} className="box">
        Box
      </div>
      <p>Width: {width}</p>
    </>
  );
}

    Here, using useEffect would cause a visible flash because the width would update after paint.

    5. üéõÔ∏è Synchronous Mutations Example

    useLayoutEffect(() => {
  elementRef.current.style.transform = 'translateX(0)';
});
    • Ensures styling updates are applied before the browser paints
    • Prevents flickering or jumpy layout

    6. üß≠ Scroll Position Adjustments

    useLayoutEffect(() => {
  containerRef.current.scrollTop = 0;
});

    Ensures scroll adjustment happens before anything is visible.

    7. ‚ö†Ô∏è Critical Rules

    ‚ùå 1. Avoid heavy work

    useLayoutEffect blocks paint ‚Äî heavy logic = jank.

    ‚ùå 2. Do not use for API calls

    Use useEffect for async side effects.

    ‚ùå 3. Do not mutate layout inside render functions

    Use useLayoutEffect instead.

    ‚ö†Ô∏è 4. Cleanup also runs before paint

    Useful for restoring previous DOM styles.

    8. üö® Common Mistakes

    1. ‚ùå Using useLayoutEffect for things that do not depend on layout
    2. ‚ùå Performing expensive operations ‚Üí leads to blocking UI
    3. ‚ö†Ô∏è Using it everywhere instead of only layout-specific places
    4. ‚ùå Misunderstanding that it runs synchronously

    9. üöÄ Performance Tips

    • Prefer useEffect unless layout measurement is required
    • Move heavy calculations outside the effect
    • Batch DOM reads separately from DOM writes
    • Avoid re-running useLayoutEffect unnecessarily (define correct deps)

    10. üí° When NOT to Use useLayoutEffect

    Avoid it when:

    • Doing API calls
    • Registering event listeners
    • Logging
    • Running async logic
    • Updating state unrelated to layout

    These should use useEffect.

    11. üì¶ Practical Patterns

    1. Auto-Focusing Without Flicker

    useLayoutEffect(() => {
  inputRef.current.focus();
}, []);

    2. Measuring Element Size Before Paint

    useLayoutEffect(() => {
  const { height } = elRef.current.getBoundingClientRect();
  setHeight(height);
});

    3. Preventing Layout Shift

    useLayoutEffect(() => {
  if (isOpen) {
    modalRef.current.style.opacity = 1;
  }
});

    12. üí° Key Takeaways

    • useLayoutEffect runs before paint
    • Use it for:
    • DOM measurement
    • synchronous DOM mutations
    • avoiding flickers

    • correcting layout before display

    • Avoid heavy logic ‚Äî it blocks rendering

    • Use useEffect for async/non-layout work
